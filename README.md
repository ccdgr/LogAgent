# LogAgent — 智能日志分析与诊断助手

基于 Go + DeepSeek 的 ReAct Agent，对生产日志进行自动化故障诊断：接收故障描述 → 异步推理排查 → 输出结构化根因分析报告。

## 快速开始

```bash
# 1. 启动依赖
mysql -e "CREATE DATABASE IF NOT EXISTS log_agent"
redis-server

# 2. 修改 config.yaml 中的数据库连接信息

# 3. 生成测试日志
go run scripts/gen_logs.go

# 4. 编译运行
go build -o logagent .
./logagent
# 访问 http://localhost:8080
```

## 迭代演进记录

以下按时间线如实记录开发过程中的关键决策、踩坑与修正。每个版本标注了「做了什么事 → 暴露了什么问题 → 如何修正」。

---

### V1.0 — 最小可行原型

**做了什么：**
- Go 单体架构，Gin + GORM + Redis
- 手写 ReAct 循环（零 AI 框架依赖）
- 三个日志工具：`list_logs` / `scan_log` / `read_context`
- 单页前端 + SSE 实时推送 Agent 推理轨迹

**设计约束（从第一天就定下的硬边界）：**

| 约束 | 原因 |
|------|------|
| 最大 5 轮迭代 | 防止 LLM 死循环刷爆 API 余额 |
| `bufio.Scanner` 流式读文件 | 无论日志多大，内存复杂度 O(1) |
| 只读工具，无 shell 执行 | Agent 不具备任何写权限 |
| 不做 Fine-Tuning | 完全依赖 System Prompt 编排能力 |

**测试日志：**纯文本格式，手工拼接了几行 `[INFO]`、`[ERROR]`、goroutine 堆栈。

### V1.1 — 日志格式升级：文本 → JSON 结构化

**暴露的问题：**

V1.0 的测试日志是随手写的纯文本：

```
[ERROR] 2026-06-16 10:30:15 panic: runtime error: nil pointer dereference
goroutine 45 [running]:
github.com/xy/LogAgent/internal/service.(*UserService).GetUserDetails(...)
```

存在三个缺陷：

1. **与真实生产环境脱节** — 正规项目都用 `slog` 或 `zap` 输出 JSON，Agent 在文本日志上训练出的排查能力无法迁移到 JSON 场景
2. **缺少关键字段** — 没有 `trace_id`，无法验证 Agent 的链路聚合能力；没有 `latency`，无法测试性能退化类故障的排查
3. **场景单一** — 只有一个 panic，没有渐进式故障（慢查询逐步恶化、连接池耗尽→级联失败），Agent 得不到充分测试

**修正方案：**

`scripts/gen_logs.go` 完全重写，输出 24 条 slog 风格 JSON 日志，每行一条：

```json
{"time":"2026-06-17T10:30:05+08:00","level":"INFO","msg":"user login success",
 "source":{"function":"internal/api.LoginHandler","file":"/app/internal/api/auth_handler.go","line":56},
 "trace_id":"trace-a1b2c3d4e5f6","uid":1001}
```

覆盖 5 个故障阶段：启动 → 正常流量 → 渐进恶化（慢查询延迟 500ms→30s）→ 严重故障（nil pointer panic + 完整堆栈）→ 部分恢复。同一 trace_id 的多条日志模拟了真实微服务全链路（middleware → service → cache → dao）。

**经验教训：**测试数据的设计决定了 Agent 能力的上限。应该先设计出覆盖所有故障模式的测试日志，再开发排查能力——而不是反过来。

### V1.2 — Agent 引擎深度评测与 5 项缺陷修复

**起因：**

用 V1.1 的 JSON 日志做了一次完整端到端评测。输入：`uid=1003 在 10:33:35 访问订单查询接口返回 500，超时 30s+`。

**评测结果：Agent 数据检索路径正确，但最终交付为空。**

Agent 的执行轨迹：

```
轮次 1: list_logs         → 发现 app.error.log          ✅
轮次 2: scan_log uid=1003  → 未发现匹配项               ❌ 浪费
轮次 3: scan_log 10:33:35  → 找到 trace-panic-xxx        ✅
轮次 4: scan_log trace_id  → 返回与轮次3相同的行         ❌ 重复浪费
轮次 5: read_context       → 获取完整堆栈               ✅
→ 达到 MaxIterations，输出"已达到最大推理轮次限制"
```

日志已全部找到，但**零分析结论输出**。

**诊断出的 5 项缺陷及修复：**

#### 缺陷 ①：轮次耗尽直接放弃，无兜底

**现象：** 5 轮用完 → 一句「已达到最大推理轮次限制」→ 用户拿不到任何诊断结果。

**根因：** 循环结束后只打了一行 log，没有对已收集的数据做任何归纳。

**修复：** 新增 `generateFallbackSummary()` 兜底机制——循环结束后强制调用一次 LLM，提示词切换为「不再调用工具，基于以上所有 Observation 直接输出诊断报告」，确保用户始终能拿到可用结论。

```go
// 轮次耗尽 → 兜底总结
fallbackResult := a.generateFallbackSummary(messages)
a.LogStep("final", fallbackResult)
```

#### 缺陷 ②：缺少结构化输出模板

**现象：** Agent 输出自由文本，用户要求的「调用链路 / 直接原因 / 底层根因 / 故障时序」四项没有保证。

**根因：** 系统提示词只说「给出诊断结论和修复建议」，没有约束输出格式。

**修复：** 提示词内置强制输出模板：

```
Final Answer:
## 一、请求完整调用链路（trace_id，uid）
## 二、500 报错分层原因
- **直接原因**：
- **底层根因**：
## 三、故障发生先后顺序与传导关系
```

#### 缺陷 ③：检索策略低效，浪费宝贵轮次

**现象：** 先搜 `uid=1003` 无结果，再搜 `10:33:35`；搜到 trace_id 后又重复搜一次同样的 trace_id。

**根因：** 提示词没有引导高效搜索策略；代码层没有去重保护。

**修复：**
- **提示词层**：新增「高效检索策略」章节——已知 uid + 时间点，组合关键词一次命中；scan_log 已返回的数据不要重复搜
- **代码层**：`scan_log` 新增去重——`seenSearches map[string]bool` 记录已执行搜索，重复搜索直接拒绝并提示换关键词

```go
searchKey := filename + "|" + strings.ToLower(keyword)
if a.seenSearches[searchKey] {
    return fmt.Sprintf("提示：关键词 '%s' 已搜索过，请直接使用之前的结果或换关键词", keyword)
}
```

#### 缺陷 ④：无收敛判定标准

**现象：** Agent 不知道什么时候该停。即使日志数据已充分，仍继续调用工具。

**根因：** 提示词没有告诉模型「什么情况下应该停止搜索、开始写结论」。

**修复：** 提示词新增收敛判定：

```
【收敛判定 — 满足任一条件立即输出 Final Answer】
- 已拿到目标 trace_id 的完整调用链路
- 已定位到 panic/error 的具体代码行号
- 已发现前置关联故障（连接池耗尽、慢查询）
- 轮次已用到第 4 轮，无论数据完备与否必须在第 5 轮输出结论
```

#### 缺陷 ⑤：不会自动归纳分析

**现象：** Agent 只负责「找日志」，不负责「分析日志」。所有 Observation 原样返回，模型没有从观察中提炼结论。

**根因：** 提示词角色定位为「工具调度器」而非「诊断分析师」。Observation 只是被动追加到消息历史，没有要求模型在每轮 Thought 中做增量归纳。

**修复思路（部分在提示词，部分待后续版本）：**
- 提示词明确 Agent 不是「搜索引擎」而是「诊断专家」，要求每轮 Thought 做增量分析
- 兜底机制中强制对全部 Observation 做归纳
- 后续版本计划：在代码层对同 trace 日志做时间排序、自动提取关键指标（latency max、error count），减少模型归纳负担

### V1.3 — 提示词误导 + 数据截断 + 兜底空洞 三项修复

**起因：**

V1.2 上线后第二次端到端评测。同一个输入：`uid=1003 在 10:33:35 访问订单查询接口返回 500，超时 30s+`。

**执行轨迹：**

```
轮次 1: list_logs                         → app.error.log             ✅
轮次 2: scan_log "uid=1003 AND 10:33:35" → 未发现匹配项              ❌ AND 不生效
轮次 3: scan_log "10:33 AND panic"       → 未发现匹配项              ❌ 同上
轮次 4: scan_log "10:33 AND error"       → 未发现匹配项              ❌ 连错三次
轮次 5: scan_log "10:33"                 → 命中但截断在第 161 行      ⚠️ 目标在 167 行
→ 兜底总结触发，但摘要基于残缺数据，内容空洞
```

**V1.2 的改进全部生效**（去重未误触发、兜底有输出），但暴露了三个更深层的问题：

#### 缺陷 ⑥（新）：提示词误解了底层匹配机制

**现象：** 模型连续三次使用了 `AND` 布尔语法，全部返回空。

**根因：** `scan_log` 底层是 `strings.Contains(line, keyword)`，做纯子串匹配。但 V1.2 提示词里的「高效检索策略」举了 `"uid=1003 AND 10:33"` 作为示例，直接误导模型认为支持布尔查询。

**修复：** 新增独立章节 **「scan_log 关键词匹配规则」**，前置声明：

```
- scan_log 使用纯子串匹配（substring），不支持 AND / OR / NOT
- 如果你输入 "uid=1003 AND error"，系统会把这整个字符串当作一个关键词
- 正确做法：只使用一个最精炼的关键词片段
```

并给出具体可操作的示例——搜 `"1003"` 而不是 `"uid=1003 AND error"`。

#### 缺陷 ⑦（新）：maxMatches=10 截断了目标数据

**现象：** 第 5 轮搜 `10:33` 命中，但返回的是行号 152-161。目标请求 uid=1003 的日志从行号 167 开始。Agent 拿到的是别人的错误日志，根本没见过目标 trace_id。

**根因：** `scan_log` 硬编码 `maxMatches=10`，而 `10:33` 这个时间段匹配的行远超过 10 条。工具静默截断，Agent 不知道自己只看到了冰山一角。

**修复（三层）：**

1. `maxMatches` 提升到 **20**（`scanMaxMatches` 常量）
2. `ScanLogFile` 返回值增加 `truncated bool`——扫描到第 maxMatches+1 条时标记截断
3. 结果中追加醒目警告：

```
⚠️ 结果已截断！仅展示了前 20 条，实际匹配更多。
请使用更精确的关键词缩小范围后重新搜索。
```

`read_context` 同理标记目标行 `>` 前缀以醒目区分。

#### 缺陷 ⑧（新）：兜底总结没有数据完整性自检

**现象：** 兜底机制触发并输出了结论，但模型在残缺数据上强行归纳，结论与事实偏离。

**根因：** 兜底提示词直接要求「基于已收集数据输出结论」，没有要求模型先评估数据是否完整。

**修复：** 兜底提示词新增 **「数据完整性自检」** 四问：

```
1. 是否找到了目标用户对应的 trace_id？
2. 是否看到了完整的调用链路？
3. 是否获取到了 panic/error 的堆栈？
4. 是否发现了前置关联故障？
```

每个否定回答对应章节标注 `[数据缺失]`。同时新增第四章节「数据完整性说明」——明确告诉用户拿到了什么、缺了什么、建议人工补充排查的方向。

### V1.4 — 最后一轮强制收敛 + 兜底上下文压缩

**起因：**

V1.3 评测：检索路径完美（一击命中 → 全链路 → 堆栈定位），但最后两轮出了问题——第 5 轮模型仍然尝试调用 `scan_log`（空参数），兜底虽然触发但输出了一个近乎空白的标题。数据完备但结论缺失。

**执行轨迹回顾：**

```
轮次1: scan "10:33:35"         → 命中 uid=1003              ✅
轮次2: scan trace-panic-xxx     → 20条 + 截断告警            ✅
轮次3: read_context 177         → 堆栈定位到 line 156        ✅
轮次4: (本轮应输出 Final Answer) → 无操作                    ⚠️ 浪费
轮次5: scan (空参数)            → 错误，被系统处理            ❌ 惯性调用
兜底:  输出「## 一、请求完整调用链路（trace_id，uid）」→ 只有标题 ❌
```

#### 缺陷 ⑨：第 5 轮仍有惯性调用工具

**现象：** 即使提示词写了「轮次已用到第 4 轮必须在第 5 轮输出结论」，模型在第 5 轮依然输出 `Action: scan_log`（甚至参数为空）。

**根因：** ReAct 循环的「Action→Observation」惯性太强。模型经过前 4 轮的条件强化（输出 Action → 获得 Observation），在第 5 轮延续了相同的行为模式。仅靠提示词约束不够，需要在代码层打断这个循环。

**修复（两层防护）：**

1. **代码层硬拦截**：循环中检测 `i == MaxIterations-1`（最后一轮），如果模型仍输出 Action，不执行工具，而是追加一条拒绝消息并 continue，给模型一次补救机会：

```go
if i == MaxIterations-1 {
    messages = append(messages, ...,
        "⚠️ 这是最后一轮，工具调用已被系统拒绝。你必须基于以上所有 Observation 直接输出 Final Answer。")
    continue
}
```

2. **第 4 轮末尾预警**：Observation 消息末尾追加 `⏰ 提醒：下一轮是最后一轮（第 5 轮），届时工具将不可用。`，让模型在第 4 轮就知道下一轮不能再调工具。

3. **提示词新增「轮次预算管理」章节**：明确前 3 轮可自由搜索、第 4 轮必须盘点、第 5 轮强制输出。

#### 缺陷 ⑩：兜底上下文溢出导致输出空洞

**现象：** 兜底触发，API 调用成功，但模型只输出了 `## 一、请求完整调用链路（trace_id，uid）`，后面全是空的。没有调用链路，没有根因分析，没有时序关系。

**根因：** V1.3 的 `generateFallbackSummary` 把完整的 5 轮消息历史（系统提示词 + 5 次 assistant 回复 + 5 次 Observation，其中 Observation 包含大段 JSON 日志）全部传给 LLM。上下文长度轻松超过 15000 字符，模型被海量原始 JSON 淹没，有效信息密度极低，只能产出空洞输出。

**修复：上下文压缩 + 独立会话**

不再传完整消息历史，而是新增 `extractKeyFacts()` 函数，从消息历史中：

1. 只提取 `Observation:` 前缀的用户消息（这些才是实际日志数据）
2. 每条截断到 2000 字符（保留首尾）
3. 总长度硬上限 8000 字符
4. 组装成精简的「事实汇总」文本块

然后启动一个**全新的独立对话**（不带沉重的历史包袱）：

```
System: "你是一个后端故障诊断专家，请基于给定的日志数据编写结构化诊断报告。"
User:   "故障现象：{originalDesc}
        === 已收集的日志数据 ===
        {compressedFacts}
        === 数据结束 ===
        请基于以上数据编写诊断报告..."
```

**关键改进：** `extractKeyFacts` 不传完整消息历史，只传压缩后的观察数据；上下文从 15000+ 字符压缩到 8000 以内，信息密度大幅提升。

### V1.6 — API 输出截断 + 逻辑冗余检测

**起因：** V1.5 评测：4 轮收敛（0% 无效调用，统计完美），但 Final Answer 只有标题没有实际分析内容。且第 3 轮 scan_log trace_id 返回的数据与第 2 轮高度重叠，统计上却算作 100% 有效。

#### 缺陷 ⑪：API 输出被截断

**现象：** 模型输出了 `### 一、请求完整调用链路...` 然后停止，二三四章节完全为空。

**根因：** `DeepSeekRequest` 结构体没有 `MaxTokens` 字段。DeepSeek API 默认 `max_tokens` 较小（通常为 1024 或 2048），而四章节诊断报告需要 2000+ tokens。模型写到一半被 API 侧截断。

**修复：** `DeepSeekRequest` 新增 `MaxTokens` 字段，所有 API 调用统一设置 `MaxTokens: 4096`。

#### 缺陷 ⑫：逻辑冗余调用被统计为「有效」

**现象：** 轮次 3 的 `scan_log trace-panic-xxx` 返回了 20 条结果，其中行 167-171 与轮次 2 完全重叠。CallStats 统计：100% 有效、0% 浪费——但人类评审一看就知道第 3 轮是浪费。

**根因：** `isObservationValid` 只看「是否返回了数据」，不关心「数据是不是新的」。

**修复（三层）：**

1. **CallStats 新增 `Redundant` 字段**：与 Invalid 分开计数，Redundant 不算严格无效但计入「浪费率」

2. **Agent 新增 `seenTraceIDs` 追踪**：`extractTraceIDs()` 从 Observation 文本中提取所有 `"trace_id":"..."`。`isTraceIDRedundant()` 判断：如果本次 scan_log 返回的全部 trace_id 都已经在前序 Observation 中出现过 → 标记为冗余

3. **Observation 追加冗余提示**：当检测到冗余时，在返回结果中追加 `💡 效率提示：此搜索包含已见过的 trace_id，属于逻辑冗余调用`，并附带实时浪费率统计

4. **提示词强化**：「不要再重复 scan 同一个 trace_id——这会被系统标记为逻辑冗余调用」

**冗余判定逻辑：**

```go
func (a *Agent) isTraceIDRedundant(obs string) bool {
    traceIDs := extractTraceIDs(obs)
    // 如果所有 trace_id 都已经在之前的 Observation 中记录过 → 冗余
    for _, id := range traceIDs {
        if !a.seenTraceIDs[id] {
            a.seenTraceIDs[id] = true  // 首次出现的 trace_id 记录下来
            return false
        }
    }
    return true
}
```

### 迭代总结：三轮缺陷递进关系

| 教训 | 说明 |
|------|------|
| **提示词的示例就是规范** | V1.2 随手写的 `uid=1003 AND 10:33` 直接导致模型系统性地用 AND 语法连错三次。示例必须与底层实现精确一致 |
| **静默截断 = 数据投毒** | 工具返回部分数据但不告知「还有更多」，模型基于残缺数据做推理，结论比没有更危险 |
| **兜底 ≠ 免责** | 兜底只是保证有输出，但不能保证输出正确。必须让模型先自检数据完整性，缺失就标注，不要硬编 |
| **迭代要缩短反馈环** | V1.2 的 AND 问题本来是 V1.1 就埋下的雷（提示词写了 AND 但工具不支持）。应该每次改完提示词立即用边界 case 验证 |

### V1.5 — 工程上线约束：硬上限 + 调用效率统计

**背景：** V1.4 建立了「5 轮内收敛」的软约束，但缺少硬性兜底和量化指标。生产环境需要明确的 SLO。

#### 新增能力

**1. 硬上限 10 轮，分阶段管理**

```
MaxIterations = 10    // 绝对硬上限
ConvergeTarget = 5    // 优化目标

第 1-3 轮: 自由搜索，正常收敛应在此完成
第 4-5 轮: 优化目标窗口，逐轮升级收敛压力（⏰ 提醒 → ⚠️ 警告）
第 6-9 轮: 仅数据严重不足时使用，每轮追问「能不能写结论」
第 10 轮: 硬上限，代码层强制拒绝工具调用，等同 V1.4 的最后一轮逻辑
```

提示词「轮次预算管理」章节更新：从「只有 5 轮」变为「最多 10 轮，目标 5 轮收敛」。每个阶段的 Observation 消息中注入实时轮次计数和对应的压力等级。

**2. Agent 调用统计（CallStats）**

新增 `CallStats` 结构体，Agent 每次工具调用后实时分类：

| 分类 | 判定规则 | 示例 |
|------|---------|------|
| 有效 | 返回了实际数据 | `搜索结果 (5 条)`、`发现文件: ...` |
| 无效 | 错误 / 未命中 / 重复 / 参数缺失 | `错误:`、`未发现匹配项`、`提示：关键词 '...' 已搜索过` |

`isObservationValid()` 按前缀匹配判定，Observation 消息末尾实时追加 `[调用统计] 总调用: N, 有效: N, 无效: N (X%)`。

**3. 兜底报告新增「调用效率统计」章节**

无论是正常收敛还是兜底触发，报告末尾都输出：

```
## 五、调用效率统计
- 总调用次数：4
- 有效调用：3
- 无效调用：1
- 无效调用占比：25%
- 质量判定：❌ 未达标（无效调用超过 20%）
- 硬上限：10 次 | 优化目标：5 次内收敛
```

**4. 无效调用 ≤20% 的质量红线**

`CallStats.IsHealthy()` 判定：`invalidRatio > 0.2` 则为不健康。兜底报告中以 ✅/❌ 直观展示。模型在每轮 Observation 中看到实时统计，形成自我修正的反馈环。

### 技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | Go 1.25 | Goroutine 异步诊断，低内存占用 |
| Web | Gin | 高性能路由，SSE 原生支持 |
| ORM | GORM | 自动迁移，快速迭代 |
| 缓存 | Redis | SSE Pub/Sub + 滑动窗口限流 |
| LLM | DeepSeek Chat | 高性价比推理 |
| 前端 | 原生 HTML5 + Bootstrap 5 | 零构建，单文件部署 |

### 关键设计决策

- **零 AI 框架**：不用 LangChain，手写 ReAct 循环。白盒可控，冷启动 <10ms
- **O(1) 内存**：`bufio.Scanner` 逐行流式扫描，分析 10GB 日志内存不涨
- **只读沙箱**：Agent 只能 list/scan/read `logs/` 目录，无 shell 无网络无写权限
- **5 轮硬上限 + 兜底**：防止 LLM 死循环，兜底确保始终有输出
